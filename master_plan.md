# master_plan.md

## 项目总计划文档

本文档是《面向小学生信息科技课程的模拟在线购物网站》项目的总路线图、实施计划和验收基线。所有参与本项目开发的 AI 代理和项目维护者应以此文档为开发指导。

***

## 1. 项目概述

### 1.1 项目目标

构建一个**简化版、可理解、可体验、可教学**的模拟在线购物网站，服务于小学信息科技课堂教学，实现三大核心场景：

| 场景 | 说明 |
|-----|------|
| 教学演示 | 教师可在课堂上演示购物网站的完整流程和功能逻辑 |
| 学生体验 | 学生可亲自操作，体验完整的在线购物流程 |
| 项目学习 | 学生可通过阅读代码理解网站结构和实现原理 |

### 1.2 目标用户

| 用户类型 | 特征描述 | 使用场景 |
|---------|---------|---------|
| 小学第二学段学生 | 3-4 年级，8-10 岁，计算机操作基础有限 | 课堂体验、项目学习 |
| 信息科技教师 | 需要清晰的教学案例和可演示的系统 | 课堂演示、教学讲解 |

### 1.3 教学场景定位

本项目不是商业电商系统，而是**教学工具**：

- **演示工具**：教师演示购物网站的工作原理
- **体验平台**：学生体验完整的购物流程
- **学习案例**：学生阅读代码理解前端开发
- **Vibe Coding 案例**：展示 AI 协同开发的工作流程

### 1.4 第一版范围概述

| 范围类型 | 内容 |
|---------|------|
| 核心功能 | 注册登录、商品浏览、购物车、订单、模拟支付、物流、评价 |
| 技术方案 | Vue 3 + Vite + Vue Router + localStorage |
| 数据方案 | 静态商品数据 + localStorage 用户数据 |
| 页面数量 | 12 个页面 |
| 模块数量 | 8 个模块 |

## 3. 总体开发目标

### 3.1 第一版目标

**核心目标：让学生能够走通完整购物流程**

```
注册/登录 → 浏览商品 → 查看商品详情 → 加入购物车 → 修改购物车 → 填写收货信息 → 提交订单 → 选择模拟支付方式 → 查看订单状态 → 确认收货 → 评价商品
```

**具体目标：**

| 目标类型 | 具体内容 |
|---------|---------|
| 功能目标 | 实现 13 项核心功能，覆盖完整购物流程 |
| 页面目标 | 完成 12 个页面，流程衔接顺畅 |
| 数据目标 | 静态商品数据 + localStorage 用户数据正常工作 |
| 体验目标 | 操作流畅、反馈清晰、适合小学生使用 |
| 教学目标 | 代码结构清晰，便于讲解和理解 |

### 3.2 第一版完成后应达到的效果

| 维度 | 预期效果 |
|-----|---------|
| 功能完整性 | 主购物流程可以完整走通，无阻塞性问题 |
| 用户体验 | 操作简单直观，反馈清晰友好 |
| 教学可用性 | 教师可在课堂上清晰演示每个功能 |
| 学生可用性 | 学生可以独立完成注册到评价的全流程 |
| 代码质量 | 结构清晰、命名规范、注释适当 |
| 可扩展性 | 模块化设计，便于后续功能扩展 |

### 3.3 不追求的目标

| 不追求的内容 | 原因 |
|------------|------|
| 真实支付能力 | 教学模拟项目，安全考虑 |
| 真实物流对接 | 教学模拟项目，复杂度过高 |
| 高并发处理 | 教学模拟项目，无此需求 |
| 复杂推荐算法 | 超出教学目标 |
| 营销功能（优惠券等） | 超出教学目标 |
| 后台管理系统 | 第一版范围外 |
| 多商家入驻 | 超出教学目标 |

## 9. 风险与控制

### 9.1 开发风险清单

| 风险类型 | 风险描述 | 风险等级 | 控制策略 |
|---------|---------|---------|---------|
| 功能蔓延 | AI 擅自增加 PRD 之外的功能 | 高 | 严格遵循 AGENTS.md 的禁止事项清单 |
| 过度设计 | 引入不必要的复杂架构或依赖 | 高 | 每引入依赖需说明理由，保持简单 |
| 结构混乱 | 代码结构不清晰，难以理解和维护 | 中 | 严格遵循目录结构和命名规范 |
| 状态不同步 | localStorage 与页面状态不一致 | 中 | 使用 composables 统一管理状态 |
| 页面能看不能用 | UI 完成但功能逻辑未实现 | 中 | 每个页面完成后必须验证功能 |
| 流程断裂 | 购物流程某环节无法继续 | 高 | 每个阶段完成后进行主流程验证 |
| 代码质量差 | 命名不规范、注释缺失、结构混乱 | 中 | 代码审查，遵循命名规范 |
| 教学不可用 | 功能可用但难以讲解或理解 | 中 | 代码可读性检查，教学场景验证 |

### 9.2 风险控制策略

#### 9.2.1 功能蔓延控制

```
控制措施：
1. 每次开发前确认任务范围
2. 开发过程中遇到新需求，先记录不实现
3. 完成后对照 PRD 检查是否超范围
4. AI 输出报告时说明是否涉及范围外功能
```

#### 9.2.2 过度设计控制

```
控制措施：
1. 每引入一个依赖必须说明理由
2. 代码复杂度检查：单个函数不超过 50 行
3. 架构检查：不引入未在技术栈中列出的技术
4. 定期审查：是否有可以简化的设计
```

#### 9.2.3 状态不同步控制

```
控制措施：
1. 所有 localStorage 操作通过 storage 模块
2. 使用 composables 统一管理状态
3. 状态变更后立即同步到 localStorage
4. 页面加载时从 localStorage 恢复状态
```

#### 9.2.4 流程断裂控制

```
控制措施：
1. 每个阶段完成后进行主流程验证
2. 验证时走完整购物流程，不跳过任何步骤
3. 发现断裂立即修复，不继续开发新功能
4. 使用主流程验收清单逐项检查
```

## 10. AI 协同开发策略

### 10.1 AI 推进方式

```
每次推进流程：

1. 接收任务
   ↓
2. 确认任务范围（对照 AGENTS.md）
   ↓
3. 阅读相关文件，理解现有结构
   ↓
4. 制定修改方案（说明要改什么、怎么改）
   ↓
5. 执行修改（小步进行）
   ↓
6. 验证修改（功能是否正常）
   ↓
7. 输出修改报告
```

### 10.2 每轮修改范围控制

| 控制项 | 限制 | 原因 |
|-------|------|------|
| 文件数量 | 不超过 5 个文件 | 保持改动可理解 |
| 代码行数 | 单文件不超过 200 行新增 | 保持改动可审查 |
| 功能数量 | 单次只完成一个小功能 | 保持改动可验证 |
| 依赖引入 | 不引入技术栈外的依赖 | 保持项目简单 |

### 10.3 修改后输出要求

每次修改完成后，AI 必须输出以下内容：

```markdown
## 修改报告

### 修改内容
- [具体修改了什么]

### 修改原因
- [为什么要这样修改]

### 影响文件
- `文件路径` - [修改说明]

### 验证方法
1. [验证步骤]
2. [预期结果]

### 下一步建议
- [建议下一步做什么]
```

### 10.4 避免 AI 一次性大改

| 措施 | 说明 |
|-----|------|
| 任务拆分 | 将大任务拆分为小任务，逐个完成 |
| 阶段验收 | 每个阶段完成后验收，再继续下一阶段 |
| 范围确认 | 开发前确认任务范围，不超范围开发 |
| 代码审查 | 每次修改后进行代码审查 |
| 用户确认 | 重要修改前与用户确认方案 |

## 14. 执行摘要

### 项目定位

这是一个**小学信息科技课程教学用的模拟在线购物网站**，不是真实电商系统。

### 核心目标

让学生走通完整购物流程：注册/登录 → 浏览商品 → 加入购物车 → 提交订单 → 模拟支付 → 查看订单 → 确认收货 → 评价商品

### 技术方案

Vue 3 + Vite + Vue Router + localStorage，前端为主，不依赖真实后端

### 开发阶段

项目初始化 → 基础设施 → 用户模块 → 商品模块 → 购物车模块 → 订单模块 → 支付模块 → 物流模块 → 评价模块 → 个人中心 → 优化完善 → 验收交付

### 关键约束

1. 不接入真实支付、真实物流、短信验证
2. 不擅自增加 PRD 之外的复杂功能
3. 不过度设计，不引入不必要依赖
4. 按小步迭代方式推进
5. 每次修改必须输出完整报告

### 验收标准

主购物流程完整可走通，所有功能正常工作，代码结构清晰，可用于课堂教学

### 风险控制

严格遵循 AGENTS.md 规范，每阶段验收后再继续，发现流程断裂立即修复
