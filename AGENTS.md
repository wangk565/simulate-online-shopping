# AGENTS.md

## AI 协同开发规范文档

本文档是《面向小学生信息科技课程的模拟在线购物网站》项目的 AI 开发代理指令文件。所有参与本项目开发的 AI 代理必须严格遵循本文档的规范与约束。

***

## 1. 项目目标

### 1.1 核心目标

构建一个**简化版、可理解、可体验、可教学**的模拟在线购物网站，用于小学信息科技课堂教学，实现"教学演示 + 学生体验 + 项目学习"三大场景。

### 1.2 教学目标

- 帮助小学生理解购物网站的基本结构
- 理解购物流程的功能逻辑
- 体验完整的交互流程
- 观察数据变化过程

### 1.3 第一版核心目标

让学生走通完整购物流程：

```
注册/登录 → 浏览商品 → 查看商品详情 → 加入购物车 → 修改购物车 → 填写收货信息 → 提交订单 → 选择模拟支付方式 → 查看订单状态 → 确认收货 → 评价商品
```

***

## 2. 目标用户

| 用户类型     | 描述                     |
| -------- | ---------------------- |
| 小学第二学段学生 | 主要是 3-4 年级学生，年龄 8-10 岁 |
| 信息科技教师   | 使用本网站进行课堂教学演示          |

## 3. 技术栈

### 3.1 核心技术

| 技术           | 版本  | 用途     |
| ------------ | --- | ------ |
| Vue          | 3.x | 前端框架   |
| Vite         | 5.x | 构建工具   |
| Vue Router   | 4.x | 路由管理   |
| localStorage | -   | 本地数据存储 |

### 3.2 数据方案

- **商品数据**：本地静态数据文件（`products.js` / `categories.js`）
- **用户数据**：localStorage
- **购物车数据**：localStorage
- **订单数据**：localStorage

### 3.3 禁止引入的依赖

- 禁止引入真实支付 SDK
- 禁止引入真实物流 API
- 禁止引入短信/邮件验证服务
- 禁止引入复杂状态管理库（如 Pinia/Vuex），第一版使用简单的响应式状态管理即可
- 禁止引入 UI 组件库，使用自定义组件保持代码可读性和教学价值

## 5. 开发原则

### 5.1 核心原则

| 原则   | 说明             |
| ---- | -------------- |
| 教学优先 | 所有设计决策优先考虑教学价值 |
| 简单清晰 | 代码结构清晰，便于讲解和理解 |
| 流程完整 | 主流程必须完整可走通     |
| 安全可控 | 不涉及真实数据和真实交易   |
| 易于扩展 | 代码结构支持后续功能扩展   |

### 5.2 开发约束

1. **不要擅自增加 PRD 之外的复杂功能**
2. **不要过度设计**：保持简单，满足教学需求即可
3. **不要引入不必要的依赖**：每引入一个依赖都要有明确理由
4. **不要把模拟项目做成真实电商系统**
5. **优先保证主流程可跑通**：功能完整性优于功能丰富性
6. **每次修改都尽量小步进行**：单次改动控制在可理解范围内
7. **修改前先理解已有结构**：尽量局部改动，避免大规模重构
8. **输出结果要说明**：改了什么、为什么改、影响了哪些文件、下一步建议做什么

## 8. UI/UX 要求

### 8.1 UI 风格约束

| 要求   | 说明                      |
| ---- | ----------------------- |
| 目标用户 | 小学第二学段学生（3-4 年级，8-10 岁） |
| 整体风格 | 简洁、明亮、友好                |
| 色彩方案 | 主色调明亮温暖，避免暗色系           |
| 按钮设计 | 大尺寸、圆角、颜色鲜明             |
| 字体大小 | 比常规网站大 20%              |
| 交互反馈 | 明显、即时、易懂                |
| 错误提示 | 友好、具体、可理解               |

### 8.2 交互设计要求

1. **按钮尺寸**：最小 44px × 44px，方便点击
2. **点击反馈**：按钮点击有明显的视觉变化
3. **加载状态**：显示加载动画或提示
4. **成功反馈**：操作成功后显示成功提示
5. **错误反馈**：操作失败后显示具体原因
6. **确认操作**：重要操作（如删除、提交订单）需二次确认
7. **空状态**：列表为空时显示友好提示

### 8.3 布局规范

- **最大宽度**：1200px，居中显示
- **响应式**：支持桌面和平板，移动端可暂不支持
- **间距**：使用 8px 基准间距系统（8px、16px、24px、32px）
- **圆角**：统一使用 8px 圆角

### 8.4 色彩建议

## 12. AI 代理工作方式约定

### 12.1 工作原则

1. **先分析再动手**：在修改代码前，先阅读和理解相关文件的结构
2. **每次只完成一个小目标**：不要一次性完成多个功能
3. **不一次性重构整个项目**：保持小步迭代
4. **不随意改动无关文件**：只修改与当前任务直接相关的文件
5. **修改后说明验证方法**：告诉用户如何验证修改是否正确

### 12.2 工作流程

```
1. 接收任务
   ↓
2. 分析任务范围（涉及哪些模块/文件）
   ↓
3. 阅读相关文件，理解现有结构
   ↓
4. 制定修改方案（说明要改什么、怎么改）
   ↓
5. 执行修改（小步进行）
   ↓
6. 输出修改报告
   - 改了什么
   - 为什么改
   - 影响了哪些文件
   - 如何验证
   - 下一步建议
```

### 12.3 修改报告格式

每次修改完成后，必须输出以下格式的报告：

```markdown
## 修改报告

### 修改内容
- [描述具体修改了什么]

### 修改原因
- [描述为什么要这样修改]

### 影响文件
- `文件路径1` - [修改说明]
- `文件路径2` - [修改说明]

### 验证方法
1. [步骤1]
2. [步骤2]
3. [预期结果]

### 下一步建议
- [建议下一步做什么]
```

### 12.4 禁止行为

1. 禁止在未理解现有代码结构的情况下进行大规模修改
2. 禁止一次性修改超过 5 个文件（除非是初始化项目）
3. 禁止在单个文件中添加超过 200 行代码
4. 禁止引入未在技术栈中列出的依赖
5. 禁止实现第一版禁止功能列表中的功能
6. 禁止修改数据存储策略（必须使用 localStorage）
7. 禁止添加复杂的架构模式（保持简单）

## 13. Bug 修复原则

### 13.1 修复流程

1. **复现问题**：先确认 Bug 可以稳定复现
2. **定位原因**：找到导致问题的代码位置
3. **评估影响**：评估修复可能影响的其他功能
4. **最小修改**：用最小的改动修复问题
5. **验证修复**：确认 Bug 已修复且未引入新问题

### 13.2 修复原则

| 原则   | 说明                    |
| ---- | --------------------- |
| 最小修改 | 只修改必要的代码，不进行额外重构      |
| 保持兼容 | 修复不能破坏现有功能            |
| 添加注释 | 在修复处添加注释说明修复的问题       |
| 记录问题 | 在代码注释中记录修复的 Bug 编号或描述 |

### 13.3 Bug 修复报告格式

```markdown
## Bug 修复报告

### 问题描述
- [描述 Bug 的表现]

### 问题原因
- [描述导致问题的根本原因]

### 修复方案
- [描述修复方案]

### 修复代码
- `文件路径` - [修改说明]

### 验证方法
1. [复现步骤]
2. [预期结果]

### 影响范围
- [评估修复可能影响的其他功能]
```

## 18. 给 AI 开发代理的执行摘要

### 项目概述

这是一个**小学信息科技课程教学用的模拟在线购物网站**，不是真实电商系统。

### 核心约束

1. **技术栈**：Vue 3 + Vite + Vue Router + localStorage
2. **数据方案**：静态商品数据 + localStorage 存储用户数据
3. **禁止事项**：不接入真实支付、真实物流、短信验证、复杂权限

### 开发原则

1. 教学优先，简单清晰
2. 小步迭代，每次只完成一个小目标
3. 不擅自增加 PRD 之外的复杂功能
4. 不过度设计，不引入不必要依赖
5. 修改前先理解现有结构
6. 修改后输出完整报告

### UI 要求

适合小学 3-4 年级学生：简洁、明亮、友好、按钮大、反馈明显

### 验收标准

主购物流程完整可走通：注册/登录 → 浏览商品 → 加入购物车 → 提交订单 → 模拟支付 → 查看订单 → 确认收货 → 评价商品

### 开发顺序

项目基础 → 用户模块 → 商品模块 → 购物车模块 → 订单模块 → 支付模块 → 物流模块 → 评价模块 → 个人中心 → 优化完善

### 输出要求

每次修改必须输出：改了什么、为什么改、影响了哪些文件、如何验证、下一步建议
